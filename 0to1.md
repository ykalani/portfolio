# 0 to 1: Retro Windows Portfolio Guide

This is the practical guide for building the portfolio from scratch into a polished retro Windows-style site.

## 1. What you are building

Think of the site as a small desktop OS in the browser:

- **Desktop**: wallpaper, icons, draggable windows, taskbar.
- **Windows**: About, Projects, Experience, Contact, Resume.
- **Launcher**: the main search bar that creates app windows from prompts.
- **State**: which window is open, focused, minimized, or closed.
- **Content layer**: the actual portfolio data rendered into the UI.
- **Backend**: optional AI/API layer for dynamic content, contact handling, or assistant features.

The desktop look is the presentation layer. The portfolio content is the product.

## 2. How the components work

### Desktop shell
The desktop is the outer layout. It owns:

- the wallpaper
- grid positioning for icons
- taskbar placement
- global keyboard shortcuts
- responsive collapse on mobile

It should never hold content logic. It only arranges the UI.

### Launcher
The launcher is the main point of the site.

It should:

- accept a plain-language app prompt
- infer the likely app type
- create a new window from a manifest
- show example prompts for quick starts
- stay focused and accessible

In the current implementation, the launcher uses a local manifest generator and a master prompt contract so a backend model can be added later without changing the UI contract.

### Desktop icons
Icons are shortcuts into the site’s main sections.

Each icon should have:

- a label
- an icon asset or CSS-drawn symbol
- a target window key
- a click/tap handler

Icons should be buttons or links, not divs.

### Window manager
This is the core interaction layer.

It should handle:

- opening a window
- bringing a window to front
- minimizing
- closing
- restoring focus

Keep this logic in a small state store so every window behaves consistently.

### Windows
Each window is a content container with a title bar, body, and controls.

Recommended structure:

- title bar
- close/minimize buttons
- content area
- optional footer/status bar

Windows should be reusable, so the same component can render About, Projects, and Contact.

### Taskbar
The taskbar shows:

- active apps/windows
- start button or menu
- clock
- optional system tray items

It should reflect state, not control it directly.

### Content system
Store all portfolio text and links in one config file or content module.

That content should include:

- bio
- experience
- project list
- contact links
- resume link

This keeps the UI separate from the content and makes edits easy.

### Generated app manifest
Every launcher request should produce a JSON manifest with:

- title
- kind
- icon
- summary
- accent colors
- window size
- panels
- actions
- notes

That contract is what keeps “any app” renderable and reliable.

### Backend
Use the backend only for things that benefit from server-side logic:

- contact form submission
- analytics events
- dynamic project data
- AI-powered assistant or content generation
- API key protection

If the portfolio is mostly static, the backend can be minimal.

### Local demo boot
The simplest way to run the demo locally is:

```bash
npm run dev
```

That starts a small Node server from the project root and serves the static files on `http://127.0.0.1:4173` or the next free port if `4173` is already taken.

## 3. Recommended architecture

### Frontend
Use a simple component-based structure, even if you are not using a framework.

Suggested split:

- `Desktop`
- `Taskbar`
- `Window`
- `IconGrid`
- `StartMenu`
- `ContentPanel`

If you use a framework, keep each of those as a component.
If you use plain HTML/CSS/JS, keep the same boundaries in code.

### State model
Keep a small state object:

- `openWindows`
- `activeWindowId`
- `zOrder`
- `isStartMenuOpen`
- `selectedIconId`

The UI should be a pure reflection of state.

### Rendering model
Use data-driven rendering:

1. read content/config
2. render icons and windows from that data
3. update state on interactions
4. rerender only what changed

Avoid hand-writing the same window markup multiple times.

## 4. How to write the code

### HTML
- Use semantic elements.
- Use buttons for actions.
- Use links for navigation.
- Keep important content readable without JS if possible.

### CSS
- Define design tokens with CSS variables.
- Build the retro look with borders, bevels, gradients, and shadows.
- Keep the theme centralized.
- Use responsive rules so mobile doesn’t feel broken.
- Respect `prefers-reduced-motion`.

### JavaScript
- Keep handlers small and explicit.
- Avoid tangled UI logic.
- Separate state updates from DOM updates.
- Use one source of truth for window definitions.
- Do not mix backend calls and rendering logic in the same function.

### Accessibility
- Every control must be keyboard accessible.
- Visible focus states are required.
- Ensure contrast stays readable.
- Use ARIA only where semantic HTML is not enough.

## 5. Best practices

### Design
- Make the retro style feel intentional, not costume-like.
- Use one strong visual identity and stay consistent.
- Keep typography readable.
- Avoid excessive animation.

### Performance
- Prefer CSS over heavy JS animation.
- Lazy-load anything nonessential.
- Keep assets small.
- Use SVG/icons instead of huge bitmaps when possible.

### Maintainability
- Keep content in one file or CMS source.
- Keep styles organized by layout, theme, and components.
- Keep interaction state in one place.
- Use naming that matches the desktop metaphor.

### Security
- Never expose API keys in the client.
- Validate contact form input server-side.
- Rate-limit public endpoints.
- Sanitize any user-generated content before rendering it.

## 6. Selecting the backend model

If you want AI features, do **not** choose a model by hype. Choose it by job.

### The practical rule
Use the cheapest model that can reliably do the task.

### Suggested split

| Task | Model type |
|---|---|
| Simple chat replies, labels, summaries | Small/fast model |
| Complex reasoning, multi-step generation | Stronger model |
| Offline or privacy-sensitive fallback | Local model |
| High-stakes content generation | Stronger model with server-side validation |

### Decision criteria
Pick a model based on:

- latency
- cost
- output quality
- tool/function calling support
- context window size
- streaming support
- deployment simplicity

### My recommendation
For a portfolio:

- use a **small fast model** for normal responses
- use a **strong model** only for complex generation
- keep a **fallback mode** that works without AI

That gives you better cost control and a more reliable experience.

### If you add an AI assistant
Keep the AI isolated behind the backend:

- frontend sends a request
- backend adds system prompts and guards
- backend calls the model provider
- backend returns a safe, trimmed response

Do not let the browser talk to the model provider directly.

## 7. Deployment approach

### Best simple path
Use a static frontend deploy plus a small API backend.

Good options:

- **Frontend**: Vercel, Netlify, Cloudflare Pages
- **Backend/API**: serverless functions, edge functions, or a small Node service

### Deployment flow
1. Build the frontend.
2. Push the static site to hosting.
3. Deploy the backend separately if needed.
4. Set environment variables in the host dashboard.
5. Test all links, windows, and API calls in production.

### Environment variables
Keep secrets in server-side env vars:

- model API key
- contact form email/token
- analytics key
- database URL if you use one

Never commit secrets to the repo.

### Hosting choice

| Need | Good fit |
|---|---|
| Fast static portfolio | Vercel / Netlify / Cloudflare Pages |
| Simple backend logic | Serverless functions |
| Persistent AI proxy or queue | Small Node backend |
| More control | Docker on a small VM |

## 8. Suggested build order

1. Build the static desktop shell.
2. Add icons and windows with dummy content.
3. Wire up open/minimize/focus behavior.
4. Style the retro Windows theme.
5. Add responsive mobile behavior.
6. Add contact form and backend if needed.
7. Add AI features last.
8. Deploy and polish.

## 9. What not to do

- Do not make the site dependent on AI to function.
- Do not hide core portfolio content behind heavy interactions.
- Do not mix content, UI, and API logic together.
- Do not use a giant framework if the site is small.
- Do not over-animate the retro look.

## 10. A good final shape

The best version of this portfolio is:

- fast to load
- keyboard accessible
- visually memorable
- easy to edit
- deployable with a single click
- optional AI-powered, but never AI-dependent

That is the right 0-to-1 target.
